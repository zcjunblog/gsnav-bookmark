# GsNav Bookmark Implementation Plan

## 职责
本文件提供给 AI 开发者作为逐步实施指令。每一步都要小而具体，每一步都必须有验证方式。本文件严禁包含代码。

## 执行前提
- 开始任一步骤前，完整阅读 `memory-bank/architecture.md`。
- 开始任一步骤前，完整阅读 `memory-bank/app-design-document.md`。
- 跨机器 / 跨会话接手时，可额外阅读 `docs/project-history-and-handoff.md` 获取历史记录与接手建议，但它不能替代 `memory-bank`。
- 每完成一个重大步骤后，更新 `memory-bank/architecture.md`、`memory-bank/app-design-document.md`、`memory-bank/implementation-plan.md`、`memory-bank/progress.md`。
- 已完成的步骤必须使用 `[x]` 标记；未完成的步骤必须使用 `[ ]` 标记。

## 范围说明
本计划先聚焦“基础版导航桌面”，不做浏览器扩展专属能力，不做复杂小组件系统，不做 AI 功能。

## Step 0. 建立 memory-bank 基线
- [x] 状态：已完成
目标：把产品、架构、技术栈、实施计划、进度记录整理完整。

执行指令：
- 创建并补全文档：`architecture.md`、`app-design-document.md`、`tech-stack.md`、`implementation-plan.md`、`progress.md`。

验证：

- `architecture.md` 含完整数据库结构。
- `app-design-document.md` 清楚说明 MVP 和非目标。

## Step 1. 收缩插件入口职责
- [x] 状态：已完成
目标：让 `gsnav-bookmark.php` 只做 bootstrap，不再承载业务逻辑。

执行指令：
- 提取插件装配类。
- 提取设置页类、页面控制器、数据库安装器。
- 保留现有 `/bookmark/` 行为不变。

验证：
- 插件激活成功，无 PHP 警告或 fatal error。
- `/bookmark/` 页面仍可打开。
- WordPress 后台设置页仍可打开。

## Step 2. 建立数据库安装器与版本管理
- [x] 状态：已完成
目标：创建 `gsnav_desktops` 和 `gsnav_items`，并写入数据库版本号。

执行指令：
- 编写安装/升级逻辑。
- 初始化全局 options 默认值。
- 约定数据库版本升级流程。

验证：
- 激活插件后能看到两张自定义表。
- `gsnav_db_version` 已写入。
- 重复激活不会报错，也不会重复创建脏数据。

## Step 3. 建立默认桌面数据读取链路
- [x] 状态：已完成
目标：前端不再依赖硬编码桌面数据，而是从服务端读取系统默认桌面。

执行指令：
- 创建桌面仓储与服务层。
- 提供只读接口，先返回系统默认桌面与项目列表。
- 将前端 `desktopApps`、`dockApps`、搜索引擎基础配置改为服务端注入或接口读取。

验证：
- 删除前端硬编码后，页面仍能正常显示桌面图标和 Dock。
- 修改数据库中的默认桌面数据后，刷新页面能看到变化。
- 未登录用户能正常读取默认桌面。

## Step 4. 实现登录用户桌面解析逻辑
- [x] 状态：已完成
目标：用户访问时优先读取自己的桌面，没有则回退到系统默认桌面。

执行指令：
- 为登录用户建立“首次访问创建个人桌面”或“从默认桌面复制个人桌面”的策略。
- 使用 `usermeta` 记录当前活跃桌面。
- 明确访客与登录用户的读取顺序。

验证：
- 两个不同用户登录后可以看到各自独立的数据。
- 新用户首次访问时能获得可用桌面。
- 访客仍只能看到默认桌面。

## Step 5. 实现书签、文件夹、Dock 的保存能力
- [x] 状态：已完成
目标：把现有前端交互变成真实 CRUD。

执行指令：
- 增加新增、编辑、删除、排序保存接口。
- 文件夹遵守“一层嵌套”规则。
- Dock 与桌面区域的排序独立保存。

验证：
- 新增链接后刷新页面仍存在。
- 新建文件夹并向其中加入链接后，刷新页面结构正确。
- 删除元素后数据库同步更新。
- Dock 排序与桌面排序不会互相污染。

## Step 6. 接入书签页登录态与用户信息
- [x] 状态：已完成
目标：未登录用户在 `/bookmark/` 可直接发起 WordPress 登录；已登录用户直接看到自己的站点账户信息。

执行指令：
- 在前端初始化数据中注入当前访问者的登录态、显示名、头像和登录/退出相关链接。
- 未登录时提供登录入口，并确保登录成功后回跳当前 `/bookmark/` 页面。
- 已登录时直接显示当前 WordPress 用户信息，不再要求二次登录。
- 明确哪些交互在未登录时应引导登录，哪些交互只读。

验证：
- 未登录访问 `/bookmark/` 时能看到登录入口。
- 点击登录后跳转到 WordPress 原生登录页，并在登录成功后回到 `/bookmark/`。
- 已登录 WordPress 用户访问 `/bookmark/` 时能直接看到自己的用户信息。

## Step 7. 实现基础设置持久化
- [ ] 状态：未完成
目标：把 `showDock`、`bgBlur`、`simpleMode`、默认搜索引擎等设置写入桌面配置。

执行指令：
- 建立桌面级 `settings_json` 读写流程。
- 前端加载时读取设置，交互后保存设置。
- 访客模式下根据后台配置决定是否仅存本地。

验证：
- 修改背景模糊后刷新页面保持一致。
- 关闭 Dock 后刷新页面仍保持关闭状态。
- 极简模式切换规则符合设计文档。

## Step 8. 实现搜索引擎模块化
- [ ] 状态：未完成
目标：把搜索引擎从单一硬编码改为可配置预设。

执行指令：
- 提供至少百度、Google、Bing 三个系统预设。
- 将默认搜索引擎持久化到桌面设置。
- 搜索跳转统一经过前端模块，不散落在模板里。

验证：
- 用户切换引擎后刷新仍保留选中状态。
- 三个引擎都能正确拼接查询词并打开结果页。
- 搜索框在空输入时不会发起错误跳转。

## Step 9. 壁纸系统服务端化
- [ ] 状态：未完成
目标：保留当前视觉体验，但去掉浏览器端暴露的 API key。

执行指令：
- 将 Unsplash / Pixabay / Pexels 请求迁移到服务端代理。
- 统一返回给前端标准化壁纸对象。
- Bing 保留为兜底来源。
- 保存当前选中壁纸到桌面设置。

验证：
- 浏览器控制台和页面源码中不再出现第三方 API key。
- 切换壁纸来源与分类后仍能加载结果。
- 选中的壁纸刷新后能够恢复。

## Step 10. 后台设置页升级
- [ ] 状态：未完成

执行指令：

- 非管理员不能访问或修改系统设置。
- 修改默认桌面后，访客打开 `/bookmark/` 能看到变化。
