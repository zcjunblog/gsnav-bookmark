## 设计来源与取舍
本插件参考了两类成熟形态：
- mTab：强调“导航 + 笔记/在线应用 + 多端同步 + 私有部署”。
- iTab：强调“组件化桌面 + 动态壁纸 + 极简模式 + 聚合搜索 + 小组件”。

WordPress 版本的取舍如下：
- 借鉴它们的桌面感、组件化、壁纸和搜索体验。
- 不复制浏览器扩展专有能力。
- 利用 WordPress 账号体系，把“多端同步”改造成“登录即同步”。
- 把产品定位成“个人导航桌面 + 站点内工作台”，而不是浏览器权限型工具。

## 产品一句话定义
一个运行在 WordPress 里的、可登录同步的导航桌面插件，用来把书签、搜索、壁纸和轻量组件统一到一个可定制首页中。

## 目标用户
- 站长本人：希望把常用链接、工具入口、壁纸和常用信息集中在一个页面。
- 登录用户：希望在多个设备上使用统一的个人导航桌面。
- 团队/会员站点用户：希望站点提供一个统一入口页，承载学习、办公、导航、快捷工具。

## 核心问题
插件要解决的是：
- WordPress 站点里没有一个“像新标签页一样好用”的统一入口。
- 现有原型只有漂亮界面，没有真实数据持久化。
- 用户希望“登录即可恢复配置”，而不是只靠浏览器本地缓存。

## 产品边界

### 本产品要做的事
- 提供 `/bookmark/` 独立桌面页面。
- 提供桌面图标、Dock、文件夹、搜索栏、时间天气、壁纸面板、极简模式。
- 允许登录用户保存自己的桌面布局与常用链接。
- 允许管理员配置默认桌面、公共壁纸策略、第三方 API 密钥。

### 本产品暂时不做的事
- 不直接接管浏览器新标签页。
- 不实现浏览器扩展级“一键收藏当前标签页”。
- 不做复杂的多层文件夹。
- 不在首版做组件市场、社交分享流、AI 卡片、笔记系统。

## MVP 范围（第一阶段必须完成）

### 1. 桌面骨架
- 全屏桌面页。
- 顶部时间、日期、农历、天气。
- 书签页登录态区：未登录显示登录入口，已登录显示当前 WordPress 用户信息。
- 中央搜索框和搜索引擎切换。
- 桌面书签区。
- 底部 Dock。
- 右键菜单。
- 极简模式开关。

### 2. 数据化书签系统
- 登录用户可读取/保存自己的桌面。
- 顶层支持链接和文件夹两种元素。
- 文件夹只支持一层子元素。
- Dock 与桌面元素可分别维护排序。

### 3. 壁纸系统
- 支持 Bing 日图。
- 支持服务端代理的 Unsplash / Pixabay / Pexels。
- 支持背景模糊调节。
- 支持记住当前选中的壁纸。

### 4. 设置体系
- 前台设置：Dock 显示、默认搜索引擎、壁纸来源、背景模糊。
- 后台设置：API keys、是否开放访客模式、默认桌面。

### 5. 同步机制
- 登录用户：保存到 WordPress 数据库。
- 访客：只读系统默认桌面；如果管理员允许，则仅保存到本地浏览器，不写数据库。

## 后续版本（明确延后）
- 小组件卡片：热搜、待办、记事本、倒计时。
- 导入导出：HTML 书签导入、JSON 配置导出。
- 公共模板：站长提供多套桌面模板。
- 多桌面切换：工作 / 学习 / 娱乐 多空间。
- 站内 Web App 模式：把站内页面封装为可打开的应用。

## 交互原则

### 原则 1：打开即用
用户进入 `/bookmark/` 页面后，不需要学习成本，第一屏就要能完成：看时间、搜索、打开常用链接。

### 原则 2：先稳定，再丰富
先把书签、文件夹、Dock、壁纸、搜索做好，再增加小组件。不要为了“看起来像 iTab / mTab”而一次性塞入过多模块。

### 原则 3：桌面优先，后台兜底
高频操作应尽量在前台完成；后台只负责系统级设置和内容兜底。

### 原则 4：访客只体验，登录才同步
WordPress 最自然的同步边界是登录态。匿名访客不应进入复杂的服务端个性化逻辑。

## 页面信息架构

### 页面 1：导航桌面 `/bookmark/`
主页面包含：
- 背景层：图片或视频壁纸。
- 头部信息：时间、日期、农历、天气。
- 账户层：未登录时显示“登录以同步”入口；已登录时直接显示 WordPress 当前用户头像、昵称和账户操作入口。
- 搜索层：搜索框 + 引擎选择。
- 内容层：桌面图标网格。
- 底部层：Dock。
- 浮层：右键菜单、文件夹弹层、设置弹层。

### 页面 2：WordPress 后台设置页
后台页面负责：
- 配置第三方 API keys。
- 开启/关闭访客模式。
- 选择默认桌面。
- 未来可增加“重建数据库”“导入默认演示数据”。

## 关键体验决策

### 登录态
- `/bookmark/` 不单独维护一套账户系统，直接复用 WordPress 登录态。
- 用户如果尚未登录，在书签页应能直接看到登录入口，而不是只看到访客默认桌面后无从同步。
- 登录入口应跳转到 WordPress 原生登录页，并在登录后回跳到当前 `/bookmark/` 页面。
- 如果用户已经在 WordPress 站点中登录，书签页应直接显示该用户的头像、昵称或显示名，不再要求二次登录。
- 书签页的“保存、同步、个性化配置”能力都依赖这一登录态展示闭环。

### 搜索
- 默认提供至少 3 个引擎预设：百度、Google、Bing。
- 桌面里始终保留“切换当前引擎”的轻交互。
- 自定义搜索引擎先放在后续版本，MVP 先支持切换和持久化。

### 壁纸
- Bing 作为零配置兜底来源，保证插件开箱即用。
- 需要 key 的第三方来源全部走服务端代理。
- 壁纸中心优先解决“稳定可选”和“切换顺滑”，而不是追求大量高级筛选。

### 书签布局
- 桌面区强调视觉浏览。
- Dock 区强调高频固定入口。
- 文件夹保留，但只做一层，避免数据结构和交互复杂度失控。

### 极简模式
- 极简模式应保留时间与搜索，把次要视觉元素降噪。
- 它是高频功能，必须可以一键切换。

## 视觉方向
- 视觉上保持当前原型的玻璃态、桌面感、壁纸氛围。
- 不照抄 mTab / iTab 的具体 UI 细节。
- WordPress 版需要更偏“站点工作台”，而不是纯浏览器玩具。
- 后续如引入组件，也要优先考虑统一节奏和信息层级，不做信息堆叠。

## 成功标准
MVP 达标的标准是：
- 用户打开 `/bookmark/` 能稳定看到完整桌面。
- 未登录用户能在 `/bookmark/` 页面直接找到登录入口，并在登录后回到书签页。
- 已登录 WordPress 用户打开 `/bookmark/` 时能直接看到自己的账户信息，而不是再次登录。
- 登录用户能保存自己的桌面项目和基础设置。
- 访客能看到系统默认桌面。
- 壁纸、搜索、Dock、文件夹都不再依赖硬编码演示数据。
- API key 不再暴露给浏览器。

## 与本地现有代码的映射
当前本地代码已具备以下可复用资产：
- 桌面整体布局与玻璃态视觉。
- 时间、农历、天气展示框架。
- 壁纸面板、Dock、文件夹、搜索框、设置弹窗的交互雏形。
- 系统默认桌面的只读链路已经接入服务端，桌面图标、Dock、搜索引擎基础配置不再完全依赖前端硬编码。
- 登录用户个性化桌面解析已经接入服务端，首次访问会获得一份从系统默认桌面复制出来的个人桌面。
- 书签、Dock、文件夹的前台保存能力已经接入服务端，当前通过最小可用的右键菜单和 prompt 交互完成真实落库。
- 书签页登录态入口与用户信息展示已经接入：未登录时显示 WordPress 登录入口，已登录时显示当前用户头像、昵称、资料页和退出入口。
- 当前仍缺失的是桌面设置持久化，以及壁纸/搜索状态持久化；下一步应直接进入 Step 7 到 Step 9，而不是回退到继续堆前端演示数据。

接下来应该做的不是重画 UI，而是把这些静态界面变成结构化、可存储、可同步的真实产品。
